# 學習反思

本文件整理 RoomRush 專題開發後的學習反思，重點放在需求規劃、Android 開發理解、團隊合作，以及後續重新整理專案時的收穫。

---

## 從生活情境出發的題目選擇

一開始在思考專題主題時，我們希望題目可以和日常生活遇到的問題有關，因此想到「空教室查詢系統」。

這個題目不是單純為了做 App 而做 App，而是來自學生在校園中的實際需求：空堂時想找地方自習、休息、討論或吃飯，但不一定知道哪些教室可以使用。

---

## 從使用者功能擴充到管理者功能

原本專案只想針對使用者查詢空教室的功能進行設計，但後來考量到內容可能較少，也希望資料能被維護，因此加入了管理者功能。

管理者功能讓系統不只是查詢工具，也能處理：

- 修改教室有課時段
- 刪除教室資料
- 新增教室
- 設定教室類型
- 查看所有教室狀態

這也讓專案從單純的查詢頁面，變成較完整的校園工具 App。

---

## 使用者介面的困難點

我主要負責使用者介面相關部分。過程中比較困難的是：

- 不同頁面之間如何傳遞資料。
- 從查詢頁點選教室後，如何把教室名稱帶到詳細頁。
- 如何依資料庫內容判斷教室類型與是否可飲食。
- 如何讓查詢條件和畫面結果連動。

這些問題讓我發現，App 開發不是只把畫面做出來就好，還需要理解畫面背後的資料流。

---

## 對 MVVM 架構的理解

在專案中，我們使用 MVVM（Model-View-ViewModel）架構來整理程式碼。

一開始看到這麼多檔案時，會覺得專案變得更複雜，例如 Activity、ViewModel、Repository、DAO、Database 都各自有不同角色。剛開始接觸時並不容易理解，也會覺得為什麼不能全部寫在同一個檔案裡。

但後來重新回頭看，會發現這樣的分工是為了讓系統更容易維護：

| 角色 | 負責內容 |
|---|---|
| Activity | 畫面顯示與使用者操作 |
| ViewModel | 畫面狀態與查詢條件 |
| Repository | 資料邏輯整理 |
| DAO | 資料庫操作 |
| Room Database | 儲存教室與課表資料 |

這也是我後續想重新閱讀專案的原因：當時雖然做出了功能，但還需要更深入理解每個檔案背後的原理。

---

## 團隊合作的收穫

這次專題也讓我學到團隊溝通的重要性。因為專案同時包含使用者功能、管理者功能、資料庫與 UI 設計，需要不斷討論：

- 功能是否符合需求。
- 操作流程是否清楚。
- 資料設計是否足夠。
- 哪些地方可能出現漏洞。
- 每個人負責的部分如何整合。

過程中需要反覆修改與協調，並不是一次就能完成。

---

## 重新整理專案後的想法

現在回頭整理 RoomRush，我覺得最重要的不是馬上大幅重構，而是先確認自己是否真正理解這份專案。

因此後續整理方向會先放在：

1. 補齊 README 與文件。
2. 重新理解使用者流程與管理者流程。
3. 拆解 Activity、ViewModel、Repository、DAO 的分工。
4. 把專案整理成可以展示的 GitHub 作品。
5. 再視能力進行低風險的小型優化。

---

## 給之後自己的提醒

- 不需要急著把專案改得很厲害，先看懂比較重要。
- 每次只追一條資料流，例如「使用者選條件 → 查詢空教室 → 顯示結果」。
- 如果要優化，應該先有功能驗收清單，避免改壞原本功能。
- 文件整理也是專案能力的一部分，因為它能讓別人理解你做了什麼。
- 這份專案可以成為鐵人賽與作品集的基礎，但內容要誠實呈現學習過程。
